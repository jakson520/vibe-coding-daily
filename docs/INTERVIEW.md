# Vibe Coding Daily 面试讲解稿

## 1. 30 秒项目介绍

Vibe Coding Daily 是我做的一个 AI 编程信息聚合与每日推荐系统。它会从 Hacker News、GitHub、Claude Code releases、Cursor changelog、VS Code updates 等来源采集 AI 编程相关内容，做去重、标签分类、中文摘要和推荐排序，最后生成每日 Top 10 日报。项目还包含历史日报、收藏、标签筛选、邮件推送框架、手动触发 token 保护和信息源状态后台。

## 2. 1 分钟项目介绍

这个项目的背景是 AI 编程工具更新很快，信息分散在 GitHub、Hacker News、工具官网和各种社区里。对正在学习 Vibe Coding、AI 编程和产品开发的人来说，每天手动筛选信息成本很高。

我把它设计成一个个人信息雷达：系统每天采集多源内容，先做清洗、去重和关键词过滤，再提取标签，使用 AI 生成中文摘要和“为什么值得看”，最后按相关度、新鲜度、来源可信度和热度综合排序，生成最多 10 条日报。前端支持今日日报、历史日报、收藏和标签筛选；后台可以查看每个信息源最近是否成功、失败原因和抓取数量。这个项目重点体现了我把一个模糊信息需求拆成可运行产品闭环的能力。

## 3. 面试官可能追问的问题与回答思路

### Q1：为什么做这个项目？

回答思路：

AI 编程领域变化很快，工具、教程、开源项目和风险讨论每天都在出现。手动订阅 RSS 或刷社区会带来很多噪音，所以我想做一个垂直主题的信息筛选系统，把“收集、筛选、摘要、推荐、归档”自动化。这个项目也贴近我自己学习 AI 编程和产品开发的真实需求。

### Q2：你的目标用户是谁？

回答思路：

目标用户是正在学习或实践 AI 编程的人，包括关注 Vibe Coding、AI Agent、Codex、Cursor、Claude Code、GitHub Copilot 的个人开发者、产品经理和独立开发者。他们不一定需要完整资讯平台，更需要每天快速知道哪些内容值得看，以及为什么值得跟进。

### Q3：和普通 RSS 阅读器有什么区别？

回答思路：

RSS 阅读器主要解决订阅和阅读问题，但它通常不理解用户关注的主题，也不会自动做跨源去重、推荐评分、中文摘要和行动建议。这个项目更像一个垂直领域的信息处理工作流：它会主动采集多个来源，过滤低相关内容，给每条内容打标签、摘要并排序，最终输出每日 Top 10，而不是把所有信息原样推给用户。

### Q4：推荐评分怎么设计？

回答思路：

当前 MVP 采用可解释的加权评分，而不是一开始就做复杂模型。主要考虑四个维度：

- 相关度：是否命中 AI coding、Cursor、Codex、AI Agent 等关键词。
- 新鲜度：越新的内容分数越高。
- 来源可信度：官方 changelog、GitHub、HN 等来源有不同 trust score。
- 热度：GitHub stars、HN points/comments 等信号。

这样做的好处是透明、容易调参，也便于后续加入用户反馈。

### Q5：AI 摘要有什么风险？

回答思路：

AI 摘要可能出现误读、过度概括或输出格式不稳定。我的处理方式是：限制摘要长度和结构，要求输出中文摘要、内容类型和“为什么值得看”；同时保留原文链接和原始摘要，避免用户只依赖 AI 输出。系统也支持本地降级摘要，当 API Key 不存在或调用失败时仍能运行。

### Q6：如何处理信息源失败？

回答思路：

信息源失败是预期情况，例如网页 403、RSS 解析失败或网络错误。项目不会让单个源失败阻塞日报任务，而是给每个 source 记录 `lastStatus`、`lastError`、`fetchedCount`、`acceptedCount` 和 `lastFetchedAt`。后台 `/admin/sources` 可以直接看到失败原因；如果某个默认源长期失败，就替换为更稳定的公开 RSS、Atom feed 或 API。

### Q7：为什么用 SQLite？

回答思路：

MVP 阶段目标是快速验证个人信息流闭环，SQLite 不需要额外数据库服务，适合本地开发、演示和单用户使用。项目通过 Prisma 定义模型，后续迁移到 Postgres 或 Supabase 时，数据模型和业务层可以较平滑地扩展。

### Q8：后续如果上线怎么做？

回答思路：

上线会优先做几件事：把 SQLite 替换为 Supabase/Postgres；使用云端定时任务或队列运行每日采集；后台加鉴权；完善 source 管理；增加用户兴趣配置；部署到 Vercel 或其他平台；邮件发送使用稳定的 SMTP/事务邮件服务。同时会增加日志和监控，确保 source 失败、邮件失败、AI 调用失败都能被发现。

### Q9：这个项目体现了你的哪些产品能力？

回答思路：

它体现了几个能力：

- 从真实痛点定义产品目标，而不是只做技术 demo。
- 能把模糊的信息需求拆成清晰流程：采集、清洗、摘要、排序、展示。
- 能控制 MVP 边界，先完成个人可用闭环，再规划云端化和个性化。
- 能考虑异常和可观测性，比如 source 状态、日志和降级策略。
- 能把技术实现转化为用户价值，例如“为什么值得看”和今日行动建议。

### Q10：为什么需要后台信息源状态页？

回答思路：

信息聚合系统的质量很依赖 source 是否稳定。如果没有后台状态，用户只会看到日报内容变少，却不知道是没有新内容还是某个源失败了。后台页可以让失败变得可见，帮助判断是否需要补 token、修 collector 或替换来源。

### Q11：如果用户兴趣不同，怎么支持个性化？

回答思路：

后续可以把关键词、标签权重和屏蔽词做成用户配置。推荐评分也可以加入用户收藏、点击、忽略等反馈信号，让系统逐步学习用户更关注工具更新、教程、开源项目还是风险提醒。

### Q12：你怎么判断这个 MVP 是否成功？

回答思路：

短期成功标准是系统能稳定生成每日 Top 10，且用户能在几分钟内判断哪些内容值得看。工程验收包括采集正常、单源失败不阻塞、摘要可降级、页面可访问、手动触发受保护、日志可追踪。产品验收则看日报是否减少用户筛选成本，是否能持续发现有价值的工具、项目和教程。
